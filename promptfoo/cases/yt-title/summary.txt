En este video descubrí cómo en Angular ya no tengo que casarme con una sola estrategia de renderizado para todo mi proyecto, sino que puedo elegirla ruta por ruta. En el frontend normalmente tenemos tres técnicas básicas: server-side rendering (SSR), client-side rendering (CSR) y static o prerenderizado. Lo interesante es que Angular, con el feature de *server routing*, me permite decidir a nivel de ruta qué estrategia usar: por ejemplo, que el home sea SSR para optimizar SEO y tiempos de carga inicial, que la página de login o de locations sea client-side porque depende de geolocalización o lógica del navegador, y que una página como “about”, cuyo contenido casi no cambia, se genere de forma estática al hacer el build.

Trabajando con una aplicación de ejemplo tipo e‑commerce, vi cómo se configura esto en el archivo `app-routes-server`. Una vez que tengo habilitado el server-side rendering en el proyecto, este archivo es el que dicta qué estrategia de renderizado se aplica a cada ruta. Por defecto, todo es server-side, pero allí puedo sobrescribir ruta por ruta: por ejemplo, `locations` marcada como `client`, y `about` marcada como `prerender`. El archivo de rutas normal (`app-routes`) sigue definiendo qué componente se asocia a cada ruta, pero es `app-routes-server` el que decide si esa ruta será SSR, client o prerender. La idea práctica es dejar el SSR como comportamiento por defecto y luego “deshabilitarlo” en las rutas que no lo necesitan, como login o ciertas secciones muy dinámicas.

Al probarlo, noté claramente la diferencia: en la página home, al ver el código fuente del navegador, todo el contenido ya aparece en el HTML inicial porque viene renderizado desde el servidor. Sin embargo, en la ruta `locations`, configurada como client-side, el HTML entregado por el servidor prácticamente no contiene contenido de la vista; es el cliente quien se encarga de renderizar todo una vez que la aplicación se ejecuta en el navegador. En cambio, la ruta `about`, al estar prerenderizada, sí aparece completamente en el código fuente, pero con la particularidad de que ese HTML se genera solamente en tiempo de build. Si quiero actualizar el contenido de “about”, tengo que modificarlo, volver a hacer `ng build` y desplegar de nuevo. Esto es ideal para contenido que casi no cambia, porque aligera la carga del servidor: este simplemente sirve un archivo estático ya preparado.

Cuando hice el build del proyecto y revisé la carpeta `dist`, pude ver cómo Angular generaba los HTML prerenderizados: por ejemplo, había un archivo específico para la página `about` con todo su contenido listo, mientras que no existía uno para `locations` porque esa ruta se renderiza totalmente del lado del cliente. Así entendí que cada ruta marcada como `prerender` termina convirtiéndose en un HTML estático que el servidor solo entrega sin volver a calcular nada. Esto abre escenarios muy potentes, especialmente para secciones tipo blog o páginas de contenido: se pueden prerenderizar todos los artículos y servirlos súper rápido, con excelente SEO y sin cargar demasiado al servidor.

Además, la documentación de Angular sobre *server routing* muestra ejemplos más avanzados: combinaciones donde la ruta principal es client, otras rutas son prerender (`about`), otras son server (`profile`) y, al final, se define que todo lo demás sea SSR. También se pueden manejar casos dinámicos más complejos, como prerenderizar múltiples páginas a partir de IDs (por ejemplo, todos los posts de un blog) mapeando esos IDs en tiempo de build para generar todos los HTML estáticos. Incluso, al trabajar en modo server, hay opciones para ejecutar lógica exclusiva en el servidor, como agregar headers especiales o llaves sensibles que nunca deberían correr en el cliente.

Lo que más me llevo es la flexibilidad: puedo tener un frontend público totalmente optimizado con SSR para mejorar SEO y rendimiento inicial, una sección interna tipo “/app” que viva completamente como SPA en client-side rendering, y páginas estáticas prerenderizadas para contenido más estable. Todo dentro del mismo proyecto Angular, sin tener que crear múltiples aplicaciones separadas. Eso sí, para habilitar este juego de estrategias es obligatorio activar primero el server-side rendering en el proyecto; a partir de ahí, puedo ir degradando ciertas rutas a `client` o `prerender`. Esto implica también cambiar el modelo de hosting, porque ya necesito un entorno capaz de ejecutar el lado servidor (por ejemplo, Node, Cloudflare, etc.), pero a cambio obtengo un control granular muy potente sobre cómo se entrega cada parte de mi sitio.

En resumen, entendí que este feature de *server routing* en Angular es pequeño en configuración pero enorme en impacto: deja de ser necesario elegir entre solo SSR o solo CSR para todo el sitio, y puedo decidir ruta por ruta la estrategia de renderizado más adecuada a mis necesidades. Ahora sé que puedo diseñar arquitecturas mucho más precisas y eficientes, equilibrando SEO, rendimiento, dinámica del contenido y complejidad de despliegue dentro de un mismo proyecto.
