En este video cuento cómo, a día de 2026, despliego una aplicación Angular con server-side rendering en la nube de Cloudflare aprovechando los cambios recientes en ambos mundos. Explico que Cloudflare ha movido su enfoque de Pages hacia Workers como servicio principal para todo lo que es Edge Computing, lo que nos da una plataforma muy potente, súper económica y rapidísima para hacer rendering en el servidor. Al mismo tiempo, Angular se ha vuelto mucho más flexible y ya no depende estrictamente de Node.js: ahora podemos usar nuevos runtimes como Deno, Bun o Cloudflare Workers gracias a la API de Angular App Engine, que permite correr nuestras apps en entornos JavaScript más ligeros y optimizados para el edge.

Empiezo creando un proyecto Angular desde cero con `ng new`, activando server-side rendering y añadiendo Tailwind. Una vez generado el proyecto, abro el código en mi IDE y agrego la dependencia de desarrollo `wrangler`, que es la herramienta que Cloudflare nos da para emular localmente sus Workers y su Edge Computing. Luego creo el archivo de configuración `wrangler.jsonc`, donde defino el nombre del servicio y, sobre todo, le indico cuál es el archivo principal del servidor y dónde están los assets. Para eso primero ejecuto un `ng build` y reviso la carpeta `dist`, donde Angular genera dos subcarpetas: `browser` y `server`. Copio las rutas relativas correctas de `server` para el main y de `browser` para los assets, y las pongo en el `wrangler.jsonc` para que todo quede bien enlazado.

A continuación me centro en un punto crítico: por defecto, Angular SSR se genera con Express y eso nos ata a Node.js, lo que no sirve para runtimes como Cloudflare Workers. Por eso duplico el archivo `server.ts` (o equivalente) y creo un nuevo archivo llamado `server.edge`. En este nuevo servidor sólo importo la API de Angular App Engine, en concreto `createRequestHandler`, y configuro la app de forma que pueda ejecutarse en un entorno que no dependa de Node. La idea es tener un servidor neutro, que funcione igual en Cloudflare, Deno, Bun y otros runtimes ligeros.

Después abro el `angular.json` y cambio las referencias del build de servidor para que apunten a este nuevo `server.edge` en lugar del servidor con Express. Y aquí viene un detalle “oculto” pero vital: en la configuración del builder debo añadir la bandera `experimentalPlatform: "neutral"`. Comento que esto casi no aparece en la documentación oficial de Angular, pero sí se ve en los esquemas del builder; sin esta opción, el bundle se genera optimizado para Node.js y no funcionaría en plataformas serverless como Cloudflare Workers. Al poner `"neutral"`, Angular construye un bundle que ya no depende de Node y se puede correr en Workers o cualquier serverless/edge platform.

Una vez ajustada la configuración, vuelvo a ejecutar `ng build` para regenerar la carpeta `dist`. Muestro cómo el archivo generado para el servidor cambia y deja atrás las dependencias de Node, quedando en un JavaScript “más puro” apto para el edge. Con el build listo, corro `npx wrangler dev` para levantar la emulación local. Al abrir la URL que me da la herramienta, verifico en el navegador, viendo el código fuente de la página, que ya se está haciendo server-side rendering: el HTML viene prerenderizado desde el servidor.

También comento un aspecto importante si mi aplicación hace peticiones HTTP: en vez de depender de cosas propias de Node, configuro Angular para usar `provideHttpClient` junto con `withFetch` (o su equivalente actual) de modo que el cliente HTTP se base en el `fetch` nativo del runtime. Esto evita problemas de compatibilidad al ejecutar en Cloudflare Workers y se integra perfectamente con SSR en el edge.

Con todo funcionando en local, paso al despliegue real. Doy por hecho que ya tengo una cuenta en Cloudflare y desde la terminal inicio sesión con `npx wrangler login`, autorizando el acceso en el navegador. Luego solo necesito ejecutar `npx wrangler deploy`, usando el mismo nombre de servicio que definí en el `wrangler.jsonc`. En un intento inicial, Cloudflare me pide especificar un `compatibility_date`, que añado en la configuración; esto sirve para que el runtime sepa desde qué fecha aplicar ciertas APIs y compatibilidades. Tras corregir eso, vuelvo a desplegar: wrangler hace el build, sube los assets del `browser` y luego el código del servidor edge.

Al final, Cloudflare me devuelve una URL pública. La abro y verifico que la aplicación está funcionando correctamente y que sigue teniendo server-side rendering al ver el código fuente. Subrayo que, si comparo el tiempo de renderizado de esta aplicación en el edge de Cloudflare con el de un servidor clásico de Node, el edge siempre gana gracias a un runtime mucho más liviano, sin costos de mantener una máquina encendida todo el tiempo y con un cold start prácticamente instantáneo.
